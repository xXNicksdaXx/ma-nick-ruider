% ==========================================================
\section{\textit{Models@run.time} Integration in GLSP}
\label{sec:impl_glsp_models_runtime}
% ==========================================================

This section describes how we integrate \textit{models@run.time} into the \gls{glsp}-based \gls{bp} variant.
The VS~Code extension contributes only the user-facing commands that start and stop the listener.
The server owns the \gls{sse} connection and filters the events it receives that are relevant to the current session.
Once active, the server translates the received messages into \gls{glsp} actions and dispatches them to the client, which updates the diagram accordingly.

% ----------------------------------------------------------
\subsection{Initiating the connection from VS~Code}
\label{subsec:impl_glsp_models_runtime_vscode_commands}
% ----------------------------------------------------------

We manage the \gls{sse} connection via three VS~Code commands contributed by the \gls{bp} build profile.
They are gated in the extension manifest, so they appear only when the \gls{bp}-profile custom editor is active.
The \texttt{start listening} command calls the server to attempt to establish the connection using a polling strategy that starts at a low frequency.
The \texttt{enable high-frequency polling} command switches to a more aggressive polling mode when needed.
We can also close the connection when it is no longer required.
All three commands are available from the default command palette in VS~Code.
Invoking any command dispatches the corresponding \gls{glsp} action through the diagram connector.

\subsection{Server-Side Wiring}
\label{subsec:impl_glsp_models_runtime_server_wiring}
% ----------------------------------------------------------

In Section~\ref{subsec:impl_overview_bp_extension}, we introduce the \texttt{BPDiagramModule} as the central wiring point for the \gls{uvl}-\gls{bp} server.
In this class, we override the \texttt{configureAdditionals} method to register the \gls{sse}-related services as singletons.
Listing~\ref{lst:bp-server-sse-binding} shows the setup of the services.
The\texttt{FMBPEventListenerService} maintains the connection to the external FMBP event stream.
\texttt{FMBPHighlightActionDispatchService} and \texttt{FMBPContextUpdateService} independently consume parsed events and translate them into the appropriate \gls{glsp} action dispatches.

\texttt{FMBPEventListenerService} implements the network layer.
On startup, the service waits for the FMBP health endpoint to become reachable before opening the event stream at \texttt{/events}.
A dedicated background thread consumes the stream, and the service automatically reconnects if the connection is lost.
Listing~\ref{lst:bp-server-sse-binding} shows the implementation.

We parse each raw payload into a \texttt{ParsedServerSentEvent}, a normalized record that exposes the event type, source path, timestamp, and structured data map.
This representation abstracts over the raw JSON lines and allows downstream services to filter and route events without coupling them to the wire format.
Crucially, the \texttt{source} field identifies the feature model from which an event originates, allowing services to discard events that do not belong to the currently open diagram.

\subsection{Translating SSE into GLSP Actions}
\label{subsec:impl_glsp_models_runtime_event_to_action}
% ----------------------------------------------------------

The listener does not manipulate the diagram directly.
Instead, services register as data listeners on the \texttt{ServerSentEventsService} and are invoked selectively for the \gls{sse} types they handle.
This publisher-subscriber design keeps each service focused on a single concern and makes it straightforward to add support for new event types without modifying existing code.

\paragraph{Visual feedback via highlight actions.}
\texttt{FMBPHighlightActionDispatchService} subscribes to \texttt{requested}, \texttt{blocked}, and \texttt{waited\_for} events.
When such an event arrives, the service resolves the corresponding \gls{glsp} element identifiers from the b-thread and b-event name embedded in the payload.
The service creates a \texttt{HighlightElementAction} if a matching element exists, and then dispatches it to the client, as seen in Listing~\ref{lst:fmbp-highlight-dispatch}.

The \texttt{HighlightElementAction} constitutes the primary visual feedback loop of the \textit{models@run.time} integration.
The client receives these actions and highlights the corresponding elements, providing an immediate visual indication of the active execution state without any polling.
The visual feedback is transient and rendered solely via CSS on the client side.
Figure~\ref{fig:highlighted-b-thread} displays the result, which is oriented at the concept in Section~\ref{subsec:vis_runtime_sse}.

\paragraph{\texttt{Env} synchronization via context updates.}
\texttt{FMBPContextUpdateService} subscribes exclusively to \texttt{context\_update} events.
Upon receipt, the service compares the incoming attribute values with those in the current model state, applies any changes, rebuilds the affected graphical node, and resubmits the updated model.
Listing~\ref{lst:fmbp-context-update} shows an excerpt of the implementation.
The client thereby receives a refreshed diagram representation.
At this stage, the changes to the \texttt{Env} feature are visible in the diagram, but not persisted in the underlying model.
This is a deliberate design choice to keep the implementation focused on runtime visualization and to avoid invoking a save action from the server.
We can still manually save the model to persist the changes if desired.
Figure~\ref{fig:context-update} displays an updated context due to the execution of a behavioral program.
